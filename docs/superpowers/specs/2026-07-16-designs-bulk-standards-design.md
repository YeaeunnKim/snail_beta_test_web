# 디자인 정보 입력 구조 개편 — 설계

- 작성일: 2026-07-16
- 대상: `src/app/dashboard/designs/`
- 상태: 승인됨 (구현 계획 대기)

## 1. 문제

베타웹에서 디자인 정보를 넣고 고치는 게 불편하다. 구체적으로:

1. **추가옵션·가격·소요시간·태그를 디자인 하나하나 들어가서 고쳐야 한다.** 실제로는 이것들이 샵 전체 또는 폴더의 기본 단위다 (예: "이달의 아트" 폴더면 연장 옵션 5만원). 폴더 단위·샵 단위로 한 번에 고칠 수 있어야 한다.
2. **고치려면 카드에서 `수정`을 누르고 스크롤을 한참 내려야 한다.** 수정 옵션이 많아 카드가 길어진다. 자주 만지는 가격·시간·태그는 카드에서 바로 만져져야 한다.
3. **디자인 전체선택 후 삭제·폴더이동이 안 된다.** 폴더명 변경, 폴더 단위 공개/비공개, 공개한 디자인 상단 노출도 없다.
4. **디자이너별로 가격·소요시간이 다른 디자인**이 카드에 전혀 드러나지 않는다.

가장 어려운 문제는 (1)에 딸린 것이다: **폴더 단위로 일괄 적용 → 디자인별로 개별 수정 → 다시 폴더 단위로 일괄 수정할 때, 개별 수정된 값을 어떻게 처리하는가.**

## 2. 제약

조사 결과 확인된 사실. 설계는 전부 이 안에서 이뤄진다.

- **백엔드 계약을 바꾸지 않는다.** (사령관 결정) `backend-context/`, `src/types/api.d.ts` 수정 없음.
- **폴더 엔티티에 기본값을 담을 필드가 없다.** `DesignFolderPublic = { id, name, sort_order, featured_month, design_count, created_at, updated_at }`가 전부다. 가격/시간/태그/옵션 기본값을 저장할 자리가 없다.
- **일괄 수정 API가 없다.** owner 쪽은 디자인 1건 PATCH만 있다. 유일한 배치성 엔드포인트 `POST /shops/me/designs/sort`는 생성 전용이다. 따라서 모든 일괄 작업은 **클라이언트가 PATCH를 N번 도는 루프**다.
- **추가옵션은 디자인마다 딸린 별도 하위 엔티티**다 (`/designs/{id}/options`). 옵션 1건씩 POST/PATCH/DELETE. 샵·폴더 레벨 옵션 템플릿 개념이 없다.
- **폴더명 변경은 계약(`DesignFolderUpdate.name`)에 이미 있고 UI만 안 붙어 있다.**
- **디자이너별 가격/시간은 이미 end-to-end로 존재한다.** `DesignUpdate.designer_prices` / `designer_durations` (둘 다 `designer_ids` 전체 목록과 함께 보내야 함), 읽기는 `DesignMe.designers[].base_price / duration_minutes`. 현재는 `designers.length >= 2`일 때만 수정 폼 안에 노출된다.
- **`page.tsx`가 2,396줄이고 인라인 컴포넌트가 14개다.** (AGENTS.md의 "약 1,218줄"은 stale.)

## 3. 핵심 개념

사장님이 외워야 할 개념은 2개뿐이다.

### 3.1 값 하나짜리 → 미리보기 후 골라서 적용

대상: 가격(`base_price`), 소요시간(`duration_minutes`), 옵션 안의 `price_delta` / `duration_delta_min`.

폴더에 "기본값"을 저장하지 않는다. **폴더 안 디자인들의 현재 값을 세서, 제일 많은 무리를 기본으로 친다.** 12개 중 9개가 50,000원이면 그게 이 폴더의 기준이고, 나머지 3개는 따로 수정된 것으로 본다.

일괄 적용을 누르면 즉시 반영하지 않고 미리보기를 띄운다:

```
폴더 「7월의 아트」 가격 일괄 변경
─────────────────────────────
새 가격:   [ − ]  55,000원  [ + ]

적용 대상 12개 중

┌ 값이 같던 9개 ───────────┐
│ ☑  50,000  →  55,000      │
└───────────────────────────┘

┌ 따로 수정된 3개  ⚠ ───────┐
│ ☐  프렌치A    70,000  유지 │
│ ☐  글리터B    45,000  유지 │
│ ☐  크롬C      80,000  유지 │
└───────────────────────────┘
   기본값: 따로 수정한 건 건드리지 않음

           [  9개에 적용  ]
```

**기본 동작은 "따로 수정된 건 건드리지 않음"** (체크 해제 상태). 사장님이 체크하면 포함된다.

이 방식의 이점:
- 저장할 상태가 없다 → localStorage 불필요 → 폰/PC 어디서 열든 같은 결과
- 백엔드 필드 추가 불필요
- 충돌을 시스템이 몰래 결정하지 않고 사장님에게 보여주고 고르게 한다

**동점 처리**: 최빈 그룹이 유일하지 않으면 (예: 6:6) 기본 그룹을 정하지 않는다. 값별 그룹을 개수 내림차순으로 나란히 보여주고, **전부 체크 해제 상태**로 시작한다. 사장님이 직접 고른다.

### 3.2 목록짜리 → 현황판에서 추가/제거

대상: 태그(`owner_tags`), 옵션의 유무(`kind` + `name` 조합).

"몇 개에 달렸는지"를 현황으로 보여주고, 넣거나 뺀다. **각 디자인의 나머지 태그·옵션은 건드리지 않는다.** 그래서 충돌 자체가 생기지 않는다 (추가/제거는 교환법칙이 성립한다).

```
폴더 「7월의 아트」  현황판
─────────────────────────────
태그 현황 (디자인 12개)

  #젤        12/12  전체       [빼기]
  #프렌치     8/12  일부  ⚠  [전체적용] [빼기]
  #여름       3/12  일부  ⚠  [전체적용] [빼기]

  + 새 태그  [                ] ⏎
    → 12개 전체에 추가됩니다

  ※ 각 디자인의 다른 태그는 그대로 유지

─────────────────────────────
추가옵션 현황

  연장   12/12  ·  5만원 · 30분    [값 맞추기]
  제거   12/12  ·  값 제각각 ⚠     [값 맞추기]
  케어    4/12  ·  3만원 · 20분 [전체적용] [빼기]
```

**옵션의 값(가격/시간)은 다시 "값 하나짜리"다.** `[값 맞추기]`를 누르면 3.1의 미리보기 화면으로 간다. 그래서 개념이 2개로 닫힌다.

**태그 상한**: 기존 `MAX_OWNER_TAGS = 10`을 지킨다. 전체적용 시 상한을 넘기는 디자인은 미리보기에서 "태그 꽉 참"으로 분류하고 건너뛴다.

## 4. 화면 구조

```
폴더 목록  ──[⚙]──> 샵 전체 현황판
   │
   ├─ 📁 7월의 아트 ──[⚙]──> 폴더 현황판
   ├─ 📁 8월의 아트 ──[⚙]──> 폴더 현황판
   └─ 📁 미분류     ──[⚙]──> 폴더 현황판   ← 미분류도 붙인다
         │
         └─ 폴더 안 디자인 목록
              ├─ [⚙ 기준]        → 현황판
              ├─ [수정 OFF/ON]   → 목록 전체에 한 번에 걸림
              └─ [선택]          → 체크박스 모드
                    └─ [폴더이동] [공개] [비공개] [삭제]
```

**현황판과 다중선택은 역할이 다르다.** 현황판은 "이 폴더 기준이 얼마지"를 정기적으로 보고 맞추는 곳. 다중선택은 "얘랑 얘 지워"처럼 가끔 하는 정리 작업.

**미분류 포함** (사령관 결정): 미분류도 폴더와 동일하게 현황판을 가진다. 데이터 조회는 `listDesigns({ unfiled: true })`. 로직은 폴더와 완전히 동일하며 `folder_id` 대신 `unfiled` 플래그만 다르다.

**수정 ON/OFF는 목록 전체 토글**이다 (카드마다가 아니라). 폴더 안 화면 헤더에 있다. OFF가 기본이고, 평소에 막 눌려서 값이 바뀌는 걸 막는다.

## 5. 디자인 카드

### 5.1 레이아웃

사진을 지금(64×64)보다 크게 하고, 사진 우측에 위에서부터 제목 → 폴더 → 가격·시간 → 태그. 앱 노출 란은 현행 유지.

```
【수정 OFF】                      【수정 ON】
┌─────────┐ 프렌치A            ┌─────────┐ 프렌치A
│         │ 📁 7월의 아트       │         │ 📁 [7월의 아트 ▾]
│  사진   │ 50,000원 · 60분     │  사진   │ [−] 50,000원 [+]
│  크게   │ #젤 #프렌치 #여름   │  크게   │ [−]   60분   [+]
└─────────┘                     └─────────┘ #젤⊗ #프렌치⊗ [+⏎]
👁 앱 노출  [공개 중]            👁 앱 노출  [공개 중]
```

- 가격 ± 단위: **1,000원** — `design-settings.tsx`의 기존 `PRICE_INPUT_STEP = 1000`을 재사용한다 (`PRICE_STEP = 5000`은 디자이너별 가격·옵션 가격용이므로 쓰지 않는다)
- 소요시간 ± 단위: **30분** — 기존 `DURATION_STEP`은 10분이라 맞지 않는다. `CARD_DURATION_STEP = 30`을 새로 추가한다
- 태그: 칩 우측 `×` 누르면 즉시 제거, 입력 후 `⏎`로 추가 — 기존 `TagInput`이 이미 정확히 이 동작이다. 그대로 재사용
- 폴더: OFF면 텍스트, ON이면 select

**기존 컴포넌트 재사용**: `design-settings.tsx`가 이미 `Stepper`(±  버튼 + 직접 입력, `step` 주입 가능)와 `TagInput`(칩 × + Enter 등록 + `MAX_OWNER_TAGS` 상한)을 export한다. 카드 인라인 편집은 새로 만들지 않고 이 둘을 쓴다. `Stepper`는 클램프를 하지 않으므로 호출부가 `clampDuration` / `clampPrice`를 적용해야 한다.

기존 인트로가(`intro_price`) 표시 — 현재 카드는 정가 취소선 + 인트로가를 보여준다. 새 레이아웃에서도 유지한다. 인트로가 편집은 카드 인라인에 넣지 않고 기존 상세 `수정` 폼에 남긴다 (카드가 다시 길어지는 걸 막기 위해).

### 5.2 디자이너별 가격·소요시간

디자인마다 기본가가 있고 디자이너가 거기서 벗어난다 — **폴더:디자인 관계와 똑같은 구조가 한 층 아래에 있는 것**이다.

전부 같으면 그냥 `50,000원 · 60분`. 다르면 범위로:

```
【수정 OFF · 디자이너마다 다른 경우】
┌─────────┐  프렌치A
│  사진   │  📁 7월의 아트
│  크게   │  50,000~70,000원 ▾ · 60~90분
└─────────┘  #젤 #프렌치

【▾ 펼침】
   기본        50,000원    60분
   김원장      70,000원    90분   따로
   박실장      50,000원    60분   기본

【수정 ON · 펼침】
   기본    [−] 50,000 [+]   [−] 60 [+]
   김원장  [−] 70,000 [+]   [−] 90 [+]  [기본으로]
   박실장   기본 따름                    [따로]
```

별도 배지 없이 숫자 자체가 "디자이너마다 다르다"를 알린다. 대부분 디자인은 전부 같을 테니 평소 카드는 현행처럼 깔끔하다.

- `designers.length < 2`면 범위·펼침 자체가 없다 (현행 `multiDesigner` 규칙과 동일)
- `[기본으로]` = 해당 디자이너의 override 제거 → 저장 시 그 디자이너를 `designer_prices`/`designer_durations` 배열에서 뺀다
- `[따로]` = override 생성, 초기값은 기본값과 동일
- 저장 규칙은 현행 유지: **기본값과 다른 override만 전송**하고, `designer_ids` 전체 목록을 함께 보낸다

### 5.3 저장 시점

- **± 스테퍼**: 연타하므로 **디바운스 800ms** 후 1회 PATCH
- **태그 ⊗ / ⏎, 폴더 select, [기본으로]/[따로]**: 즉시 PATCH
- **낙관적 갱신**: 화면을 먼저 바꾸고, 실패하면 되돌리면서 토스트로 알린다
- 카드의 기존 자기 폴링(`refetchInterval: 3000`, `ai_analysis_status`가 pending/in_progress일 때)과 낙관적 갱신이 충돌하지 않게, 인라인 편집 중인 필드는 폴링 결과로 덮어쓰지 않는다
  - 이미지 처리 폴링 분기는 `3a15ac9`에서 제거됐다(자동 VM 호출 폐지). 폴링은 AI 분석만 본다

## 6. 일괄 적용의 현실

일괄 API가 없으므로 **PATCH를 N번 돈다.** 50개 폴더면 50번이다.

```ts
// _lib/apply.ts
async function applyToMany<T>(
  targets: T[],
  fn: (t: T) => Promise<void>,
  onProgress: (done: number, total: number) => void,
): Promise<{ ok: T[]; failed: { target: T; error: unknown }[] }>
```

- **동시 실행 3~4개**로 제한 (모바일 웹 + 백엔드 부하 고려)
- 진행률을 보여준다
- **중간 실패해도 롤백하지 않는다.** PATCH는 트랜잭션이 아니라 되돌릴 방법이 없다. 대신 "43개 완료, 7개 실패" + 실패 목록 + `[재시도]`를 띄운다. 재시도는 실패한 것만 다시 돈다.
- 태그 하나 추가도 디자인마다 `owner_tags` 배열 **전체**를 다시 보내야 한다 (부분 추가 API 없음). 그래서 태그 1개 추가 = PATCH N번.
- 옵션 추가/제거는 디자인마다 POST/DELETE 1번씩.

**느리다. 이건 백엔드에 일괄 엔드포인트가 생기기 전엔 못 고친다.** 나중에 갈아끼울 수 있게 `applyToMany()` 한 군데로 몰아둔다.

`_lib`은 순수 함수라 UI 없이 검증 가능하다.

```ts
// _lib/standards.ts
/** 값별로 묶어 개수 내림차순 정렬. 최빈이 유일하면 base, 아니면 base=null */
function groupByValue<T>(
  designs: Design[],
  pick: (d: Design) => T,
): { base: { value: T; designs: Design[] } | null; groups: { value: T; designs: Design[] }[] }

/** 태그별 커버리지 */
function tagCoverage(designs: Design[]): { tag: string; count: number; total: number }[]

/** (kind, name)별 커버리지 + 값 일치 여부 */
function optionCoverage(designs: Design[]): {
  kind: DesignOptionKind
  name: string
  count: number
  total: number
  priceDelta: number | 'mixed'
  durationDelta: number | 'mixed'
}[]
```

**부품 → 조립 순서를 지킨다.** 1단계에서 만드는 ± 스테퍼와 태그칩을 3단계 현황판이 그대로 재사용한다. 그래서 헛수고가 없다.

## 8. 단계

### 1단계 — 카드
- 파일 쪼개기 (위 구조로)
- 카드 레이아웃 변경 (사진 크게, 우측 제목-폴더-가격/시간-태그)
- 수정 ON/OFF 토글 (목록 전체)
- 인라인 편집: 가격 ±1,000 / 시간 ±30 / 태그칩 ⊗·⏎ / 폴더 select
- 디자이너별 범위 표시 + 펼침 + 인라인 편집 + [기본으로]/[따로]
- 디바운스 저장 + 낙관적 갱신 + 실패 롤백

### 2단계 — 목록
- 다중선택 모드 (전체선택/해제)
- 선택 → 삭제 / 폴더이동 / 공개 / 비공개
- 폴더명 변경 (`DesignFolderUpdate.name` — 계약에 이미 있음)
- 공개한 디자인 폴더 상단 우선 정렬

### 3단계 — 일괄
- `_lib/standards.ts` 최빈 그룹·커버리지 집계
- `apply-preview.tsx` 값 하나짜리 미리보기
- 폴더 현황판 (미분류 포함)
- 샵 전체 현황판

기존 `수정` 상세 폼은 그대로 둔다. 사진 교체, 설명, 인트로가 편집은 거기가 맞다.

## 9. 결정된 세부 사항

**공개 우선 정렬**: `visibility === 'active'`를 앞으로, 나머지는 기존 순서 유지. 클라이언트 정렬 (`FolderDesigns`가 `collectAll`로 전부 가져오므로 가능).

**폴더 단위 공개/비공개**: 별도 버튼을 만들지 않는다. 다중선택 → 전체선택 → `[공개]`로 충족된다. `changeVisibility`를 `applyToMany`로 N번 돈다.

**샵 전체 현황판의 로딩**: 최빈값을 세려면 전 디자인이 필요하다. 지금 목록은 50개씩 커서 페이징이므로 `collectAll`로 전부 가져온다. 디자인이 수백 개면 첫 로딩이 굼뜨다. 폴더 단위는 문제없고 샵 전체만 해당된다. 로딩 중에는 진행 표시를 띄운다.
